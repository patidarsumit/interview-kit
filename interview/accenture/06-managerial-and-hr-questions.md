# 06. Managerial And HR Questions

Final interviews often include behavioral and fit questions.

Use STAR:

```text
Situation
Task
Action
Result
```

## Describe A Challenge

Use a real project issue.

Structure:

```text
We had <problem>.

```

## Conflict With Team Member

Answer:

```text
I try to keep discussion technical and outcome-focused. In one case, there were different opinions about implementation. I compared options based on maintainability, performance, timeline, and testing. We aligned on the simpler approach and documented the decision.
```

## Handling Tight Deadline

Answer:

```text
I clarify priority, split must-have and nice-to-have, identify risks early, communicate progress daily, and avoid compromising critical quality like security and major test coverage. If needed, I suggest phased delivery.
```

## Client Requirement Changes

Answer:

```text
I first understand the business reason, then assess impact on UI, API, testing, and timeline. I communicate tradeoffs and update sprint scope through the proper Agile process.
```

## Weakness

Safe answer:

```text
Earlier I used to spend extra time perfecting implementation details. I have improved by aligning first on business priority, delivering iteratively, and taking feedback early.
```

## Biggest Challenge You Faced

Use this structure:

```text
One challenging situation was when we had <production issue / performance issue / release issue>.

The impact was <users affected / page slow / release blocked>.

I first analyzed <logs / network calls / Angular component flow / API response / browser performance>.

Then I found that <root cause>.

I fixed it by <technical solution>, added <test / monitoring / review>, and coordinated with <QA/backend/product>.

The result was <improved performance / issue resolved / release completed / fewer defects>.
```

Example:

```text
One challenging situation was a page performance issue where a dashboard was loading slowly because multiple widgets were calling APIs independently and some template logic was expensive.

I analyzed the network tab, component lifecycle, and repeated API calls. I found that some calls were triggered multiple times and a few calculations were happening directly in the template.

I moved repeated calculations into component logic, used RxJS sharing where needed, optimized change detection, and coordinated with backend to combine a few payloads.

After that, page load improved and the issue became easier to monitor and maintain.
```

## Architecture Design In Projects

```text
At a high level, we followed a modular Angular architecture.

Features were separated by business modules. Shared reusable components, pipes, directives, and utilities were kept in shared areas. Core services like authentication, interceptors, route guards, configuration, and logging were kept separately.

For API calls, services handled HTTP communication and components focused mainly on presentation and interaction logic. We used reactive forms for complex forms and RxJS for async flows.

The goal was to keep the application maintainable, testable, and scalable as new features were added.
```

## How Do You Ensure Code Quality?

```text
I focus on code quality from multiple angles.

First, I keep components small and reusable, separate business logic into services, and use clear TypeScript types.

Second, I follow code reviews, linting, unit tests, and meaningful naming conventions.

Third, I check performance, accessibility, and security basics like avoiding unsafe HTML, validating forms, and handling errors properly.

In CI/CD, tools like SonarQube quality gates can help track bugs, vulnerabilities, code smells, duplication, and test coverage before code is merged or deployed.
```

## SonarQube Quality Gate

```text
A SonarQube quality gate is a set of rules that decides whether code is acceptable to move forward in the pipeline.

It usually checks things like bugs, vulnerabilities, security hotspots, code smells, duplicate code, maintainability, and test coverage.

If the quality gate fails, the team should review the issue, fix critical problems, or discuss exceptions only when there is a valid reason.

In my work, I treat quality gates as an early warning system, not just a formality.
```

## WCAG And Accessibility

```text
WCAG stands for Web Content Accessibility Guidelines. It helps make web applications usable for people with disabilities.

In Angular UI development, I focus on semantic HTML, keyboard navigation, proper labels, focus management, color contrast, accessible error messages, and ARIA only where needed.

For example, a custom dropdown should support keyboard navigation, visible focus, screen reader labels, and correct expanded/selected states.

Tools like Axe DevTools can catch many accessibility issues, but manual keyboard testing is also important.
```
